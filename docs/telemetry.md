# Telemetry in Speedy Git

Speedy Git can send **anonymous, aggregate-only usage statistics** to help the
maintainer decide which features to invest in, which platforms to support, and
which errors to fix. Collection is completely silent (no prompts or
notifications, ever), fire-and-forget (it never affects performance), and
governed by a **dual consent** rule described below.

## How to opt out

Telemetry is sent **only when both** of these settings allow it — turning
either one off stops all collection immediately, no restart needed:

1. **VS Code's global telemetry level** — `telemetry.telemetryLevel`. Speedy
   Git honors it automatically, including live changes.
2. **The extension setting** — `speedyGit.telemetry.enabled` (default: on).
   Find it by searching "telemetry" in Settings; it appears alongside VS
   Code's own telemetry settings.

The extension setting can only further restrict the global choice — it can
never override a global "off".

Development builds, debug (F5) sessions, test runs, and any build without a
collection destination configured send nothing at all.

## Passive transparency

- A dedicated **"Speedy Git Telemetry"** output channel (View → Output) logs
  one status line saying whether telemetry is enabled or disabled (and why),
  plus one line for every event actually sent — so you can inspect exactly
  what leaves your machine. The channel never opens itself.
- A machine-readable event manifest ships at the extension root
  (`telemetry.json`) and surfaces in `code --telemetry`.

## What we collect

Every event name and property value comes from a **closed, reviewable catalog**
(`shared/telemetry.ts`); free-form strings are not representable, and the
collection funnel re-validates every report against the catalog before
recording. The full catalog:

| Event | Properties (closed sets) | Measurements | When |
|---|---|---|---|
| `activate` | `hasMultiRoot` (`true`/`false`) | `activationMs`, `repoCount` | once per session |
| `panelOpened` | `trigger` (`command`/`scmButton`) | — | when the graph panel is created |
| `operation` | `operation` (allowlisted id, e.g. `mergeBranch`), `outcome` (`success`/`error`), `errorCode` (standardized enum, error only) | `durationMs` | per user-initiated git operation |
| `uiInteraction` | `surface`, `action` (fixed ids, e.g. `commitMenu`/`cherryPick`) | — | per tracked click |
| `dialogOutcome` | `dialog` (fixed id), `outcome` (`confirmed`/`cancelled`) | — | per dialog close |
| `settingsSnapshot` | option choices as enum strings (date format, avatars, tags, remote branches, toolbar options, status bar text, signature column) | `batchCommitSize`, `overScan` | once per session |
| `perf` | `kind` (`initialLoad`/`topology`), `commitCountBucket` (`<=500` … `>10000`) | `durationMs` | once per initial load |
| `error` | `area` (functional area id), `errorCode` (standardized enum) | — | background failures only |

Each event also carries standard environment context: editor product name and
version (`common.appName`, …), host kind, UI kind, OS/platform, extension
version, and an **anonymized random machine identifier** (generated by VS
Code, not traceable to a person) used only for approximate distinct-user
counts.

High-frequency activity is **never** instrumented: scrolling, hovering,
keystrokes, row selection, commit loading/pagination, detail fetches, avatar
and signature lookups, and auto-refresh produce no events.

## What we NEVER collect

- Repository names, paths, or remote URLs
- Workspace folder names
- Branch, tag, stash, or worktree names
- Commit hashes, commit messages, or diffs
- Author names, emails, or avatars
- Git configuration values
- File names or paths
- Raw git output or exception messages (only standardized error codes)
- Search or filter text and dates
- Anything you type into any input, ever

Exact repository magnitudes (e.g. commit counts) are reported only as coarse
buckets so they cannot fingerprint a user.

## Where the data goes

Events are sent through Microsoft's official
[`@vscode/extension-telemetry`](https://github.com/microsoft/vscode-extension-telemetry)
module to a maintainer-owned Azure Application Insights resource, and are used
solely in aggregate to answer product questions (feature usage ranking, error
frequency by code, platform distribution, load-performance distribution).
