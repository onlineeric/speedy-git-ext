# Implementation Plan: Fast-forward Local Branch from Remote

**Branch**: `043-fast-forward-branch` | **Date**: 2026-05-08 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/043-fast-forward-branch/spec.md`

## Summary

Add a `Fast-forward Local Branch from Remote` action to the local-branch context menu in the graph webview. When the user confirms the dialog, the extension runs `git fetch <remote> <branch>:<branch>` against the auto-picked remote, advancing the local branch ref to the remote tip without checkout and without touching the working tree.

**Technical approach**: Reuse the existing `ConfirmDialog` primitive (already used for tag/remote-branch deletion) — no new dialog component is justified, since the clarifications removed all dialog options (no force, no remote picker). On the backend, add a focused `fastForwardFromRemote(remote, branch)` method to `GitBranchService` that wraps the refspec form of `git fetch`. Wire it through one new RPC message pair (`fastForwardLocalBranch` request → reuse existing `success` / `error` responses + `sendInitialData()` graph refresh). Add a single command-builder function and a single `BranchContextMenu` entry guarded by `isLocalBranch && !isCurrentBranch`.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode with `noUnusedLocals`, `noUnusedParameters`, `noImplicitReturns`)
**Primary Dependencies**:
- Backend: `vscode` (host API), Node.js 18 child_process via existing `GitExecutor`
- Webview: React 18, Zustand, `@radix-ui/react-context-menu`, `@radix-ui/react-alert-dialog`
**Storage**: N/A — operation is stateless; relies on the loaded `branches: Branch[]` already in the Zustand store
**Testing**: Vitest (unit) for `GitBranchService.fastForwardFromRemote`, `gitCommandBuilder.buildFastForwardLocalBranchCommand`, and any picker-resolution helper. Manual smoke test via VS Code "Run Extension" launch.
**Target Platform**: VS Code 1.80+ on macOS / Linux / Windows; webview runs in Electron's sandboxed Chromium
**Project Type**: VS Code extension — backend (Node.js extension host) + frontend (React webview) communicating via `postMessage` RPC
**Performance Goals**:
- Context-menu open: no synchronous git call, no extra extension-host round-trip — purely client-side decision from existing store data
- Fast-forward operation: bounded by remote network latency; existing 60s timeout pattern used in `pull` is reused
**Constraints**:
- MUST NOT change the currently checked-out branch
- MUST NOT modify the working tree
- MUST NOT create or apply a stash
- MUST NOT block the menu open on any extension-host RPC
- Visibility filter: only on `isBranch && isLocalBranch && !isCurrentBranch` (using existing flags in `BranchContextMenu`)
**Scale/Scope**: ~5 source-file touches; ~80 LOC backend, ~120 LOC webview, ~50 LOC tests. No new dependencies. No new dialog component, no new shared types beyond the RPC message pair.
