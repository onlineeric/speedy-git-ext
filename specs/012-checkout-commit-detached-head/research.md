# Research: Checkout Commit (Detached HEAD)

**Branch**: `012-checkout-commit-detached-head` | **Date**: 2026-03-16

## Finding 1: Git Command for Detached HEAD Checkout

**Decision**: Use `git checkout <full-hash>` (no branch argument).
**Rationale**: Git natively enters detached HEAD state when you checkout a commit hash directly. No flags needed. Output goes to stderr but the process exits 0 on success.
**Alternatives considered**: `git switch --detach <hash>` — newer API (git 2.23+), but `git checkout` is universally available and already used throughout `GitBranchService`. Using `checkout` keeps the code consistent.

## Finding 2: Existing Stash-and-Checkout Pattern

**Decision**: Mirror the branch checkout pattern exactly: `checkoutCommit` request → dirty-tree check → `checkoutCommitNeedsStash` response if dirty → `stashAndCheckoutCommit` request from UI → stash then checkout.
**Rationale**: The branch checkout flow (`checkoutBranch` → `checkoutNeedsStash` → `stashAndCheckout`) is already established. Mirroring it for commit checkout keeps the mental model consistent and reuses the same `isDirtyWorkingTree` utility and `GitStashService.stash()` method.
**Alternatives considered**: A single round-trip message that always stashes if dirty — rejected because it would stash without user consent, which is unexpected behavior.

## Finding 3: Hash Validation

**Decision**: No `validateRefName` call needed for commit hashes — hash is sourced directly from the in-memory `Commit` object (never from user input), so it is already a trusted 40-character hex string.
**Rationale**: `validateRefName` guards against shell injection in branch/tag names that could contain special characters. Commit hashes are hex strings generated by git itself. Cross-referencing: other operations that use `commit.hash` directly (e.g., `cherryPick`, `revert`, `dropCommit`) do not validate the hash.
**Alternatives considered**: Adding validation anyway as defense-in-depth — unnecessary given the internal trust boundary.

## Finding 4: Frontend State for Pending Commit Checkout

**Decision**: Add `pendingCommitCheckout: { hash: string } | null` to `graphStore.ts`, parallel to the existing `pendingCheckout: { name: string; pull?: boolean } | null`.
**Rationale**: The stash dialog must be shown outside the context menu lifecycle (the context menu may unmount after selection). Storing pending state in Zustand ensures the dialog remains visible. The pattern is identical to `pendingCheckout` for branch checkouts.
**Alternatives considered**: Keeping all state local to `CommitContextMenu` — would require the menu to stay mounted, which Radix UI doesn't guarantee after `onSelect` fires.

## Finding 5: Menu Item Placement

**Decision**: Place "Checkout this commit" at the top of the commit context menu (before "Create Branch Here..."), with a separator after. Disabled when `isOperationInProgress`.
**Rationale**: Checkout is a primary navigation action and deserves top placement. `isOperationInProgress` guard is already computed in `CommitContextMenu` and applies to Revert and Drop Commit per FR-010.
**Alternatives considered**: Placing it near the bottom — less discoverable for the primary use case.

## Finding 6: Short Hash Display

**Decision**: Use `commit.abbreviatedHash` (already 7-character abbrev hash on every `Commit` object) in the confirmation dialog message.
**Rationale**: `abbreviatedHash` is the standard abbreviated hash already displayed in commit rows throughout the UI. Using it in the dialog gives users a visual anchor to match what they see in the graph.
**Alternatives considered**: Showing the first 7 characters of `commit.hash` manually — redundant when `abbreviatedHash` already exists on the type.
