# RPC Message Contracts: Compare Refs (A vs B)

**Feature**: `042-compare-refs`

This document defines the RPC messages added to `shared/messages.ts` for the compare feature.
All messages flow over the existing webview ↔ extension `postMessage` channel and follow the
established `RequestMessage` / `ResponseMessage` shape.

Existing types referenced: `SlotValue`, `CompareMode`, `CompareResult`, `FileChange`,
`GitError` (`shared/types.ts`, `shared/errors.ts`).

---

## Request: `compareRefs` (webview → extension)

```ts
| {
    type: 'compareRefs';
    payload: {
      a: SlotValue;
      b: SlotValue;
      mode: CompareMode;       // 'two-dot' | 'three-dot'
      requestId: string;       // opaque, generated by the webview; echoed in the response
                               // and used to match cancellation
    };
  }
```

**Semantics**:

- The backend resolves `a` and `b` to commit-ish strings via internal `resolveSlot(SlotValue)`:
  - `workingTree` → no second arg (working-tree comparison form)
  - `head` → `'HEAD'`
  - `branch` → `name` (with optional `remote/name` form)
  - `tag` → `name`
  - `commit` → resolved hash
  - `expression` → raw `text`
  - `emptyTree` → `EMPTY_TREE_HASH`
- Backend issues `git diff --name-status -z` and `git diff --numstat -z` per Decision 1 in
  research.md.
- If `mode === 'three-dot'` and the two refs share no merge base, the backend retries with
  `two-dot` and sets `fellBackToTwoDot: true` (FR-012).
- If either slot is `workingTree` and `mode === 'three-dot'`, the backend treats it as a
  client bug and returns an `error` response with code `INVALID_INPUT` (the UI is supposed to
  prevent this per FR-011; treat backend as defense-in-depth, not as a redundant validator).
- Backend tracks the active `compareRefs` request via a single `AbortController`. A new
  `compareRefs` request implicitly aborts the previous one (latest-wins).

**Errors** (delivered as standard `error` response):

| Condition | `GitError.code` | Inline message shown to user |
|---|---|---|
| Slot value cannot resolve (e.g., deleted branch) | `COMMAND_FAILED` | `Unknown ref: <slot label>` |
| Underlying git diff fails | `COMMAND_FAILED` | git's stderr, surfaced as inline panel error (FR-034) |
| Cancelled by user | `CANCELLED` (new code) | (no toast — UI just clears loading per FR-025b) |
| Repo missing | `NOT_A_REPOSITORY` | existing handler |
| Timeout (30s default) | `TIMEOUT` | existing handler |

## Response: `compareResult` (extension → webview)

```ts
| {
    type: 'compareResult';
    payload: {
      requestId: string;       // echoes the request
      result: CompareResult;   // shape defined in data-model.md §4
    };
  }
```

**Semantics**:

- The webview MUST ignore a `compareResult` whose `requestId` is older than the most recent
  request it dispatched (latest-wins also at the UI layer).
- On receipt, the store transitions:
  `comparePanelUI.loading = false; compareResult = payload.result; aResolvedHash/bResolvedHash
  populated from result`.

## Request: `cancelCompare` (webview → extension)

```ts
| {
    type: 'cancelCompare';
    payload: {
      requestId: string;       // identifies which in-flight request to cancel
    };
  }
```

**Semantics**:

- Backend looks up the active `AbortController` keyed by `requestId`. If matched, calls `abort()`
  on it. The in-flight `compareRefs` resolves with a `CANCELLED` `GitError` and the standard
  `error` response carries that code.
- If `requestId` doesn't match (already completed / superseded), the cancel is a no-op.
- No dedicated `cancelCompareAck` response — the `error` response with `CANCELLED` code is the
  ack.

## Backend service contract: `GitDiffService.compareRefs`

```ts
class GitDiffService {
  async compareRefs(
    refA: string,
    refB: string | null,                 // null when comparing against working tree
    mode: CompareMode,
    abortSignal?: AbortSignal,
  ): Promise<Result<CompareResult, GitError>>;
}
```

**Behavior**:

1. Validate `refA` (and `refB` if non-null). For raw 40-hex hashes, reuse `validateHash`. For
   other forms, no client-side validation (git rejects on resolve).
2. Build `git diff` args:
   - Two-dot: `['diff', '--name-status', '-z', refA, refB]` (or `['diff', '--name-status', '-z', refA]`
     when `refB === null`).
   - Three-dot: `['diff', '--name-status', '-z', `${refA}...${refB}`]`.
3. Run `--name-status` and `--numstat` in parallel via `GitExecutor`, both passing `abortSignal`.
4. If three-dot fails with stderr matching "no merge base" (or `git merge-base` returns empty),
   retry with two-dot and set `fellBackToTwoDot = true`.
5. Parse outputs with the existing `parseDiffNameStatus` and `applyNumstatToFiles` helpers
   (already in `GitDiffService`).
6. Resolve final hashes (`git rev-parse <ref>`) for the `aResolvedHash` / `bResolvedHash` fields.
   Working-tree side resolves to `null`; `emptyTree` resolves to `EMPTY_TREE_HASH`.

**Returns**: `Result<CompareResult, GitError>` per Constitution Principle III.

## Backend service contract: `GitExecutor` extension

```ts
export interface GitExecOptions {
  args: string[];
  cwd: string;
  timeout?: number;
  env?: Record<string, string>;
  abortSignal?: AbortSignal;            // NEW
}
```

`GitExecutor.execute` MUST:

- Register a one-shot listener `abortSignal.addEventListener('abort', () => gitProcess.kill())`.
- Resolve with `err(new GitError('Cancelled', 'CANCELLED', cmdString))` when the abort fires
  before the process exits.
- Treat `abortSignal` as already-aborted at call time — short-circuit and return `CANCELLED`
  before spawning.

## Per-file diff contract

The webview triggers VS Code's diff editor by calling the existing `openDiff` RPC, but with the
two endpoints' `git-show://` URIs (or a working-tree file URI for the working-tree side).

The existing message:

```ts
| { type: 'openDiff'; payload: { hash: string; filePath: string; parentHash?: string; status?: FileChangeStatus } }
```

is **not sufficient** for compare because both sides may be arbitrary refs. Add a sibling
message:

```ts
| {
    type: 'openCompareDiff';
    payload: {
      filePath: string;
      aHash: string | null;       // null = working tree (file system URI used)
      bHash: string | null;       // null = working tree
      status: FileChangeStatus;   // for the title decoration
      title: string;              // panel title, e.g., "main → feature/x · path/to/file"
    };
  }
```

The extension-host handler builds the two URIs:

- `null` → `vscode.Uri.file(workspaceRoot/filePath)`
- non-null → `vscode.Uri.parse(`git-show://${hash}/${filePath}?${filePath}`)`

…then calls `vscode.commands.executeCommand('vscode.diff', leftUri, rightUri, title)`.

`GitShowContentProvider` requires no changes — it already handles arbitrary commit hashes,
including `EMPTY_TREE_HASH`, via `git show <hash>:<path>`.

## Compile-time exhaustiveness

Both new request types (`compareRefs`, `cancelCompare`, `openCompareDiff`) and the new response
type (`compareResult`) MUST be added to the `REQUEST_TYPES` / `RESPONSE_TYPES` const maps in
`shared/messages.ts` so the existing exhaustive-narrowing TypeScript check fires if a future
edit drops one.
