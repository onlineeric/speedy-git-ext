# 041 — Startup Performance Fixes

## Problem

On a slower PC, the extension takes 4-7 seconds before the graph is displayed. Compared to Git Graph which loads in 2-3 seconds, this is too slow for a "Speedy Git" app.

Trace logging revealed three root causes that compound to create the delay.

## Root Cause Analysis (from trace data)

**Test environment**: Slower work PC, repo with ~1000 commits, 405 branches, 80 authors, NO submodules.

### Timing breakdown (initial load)

| Operation | Duration | Notes |
|---|---|---|
| getCommits (1000) | 719ms | OK |
| getUncommittedSummary | 615ms | OK (internally 4+3 parallel git processes) |
| getRemotes | 589ms | OK |
| getRevertState | 637ms | OK |
| listWorktrees | 692ms | OK |
| getStashes | 795ms | OK |
| getAuthors | 1007ms | `git log --all --format=%an%x00%ae` |
| getBranches (405) | 1194ms | `git branch -a --format=...` |
| **getSubmodules** | **2650ms** | **BOTTLENECK** — `git submodule status` is extremely slow under process contention |
| **Total (wall clock)** | **2821ms** | Bounded by slowest = getSubmodules |

Without `getSubmodules`, `Promise.allSettled` would complete at ~1200ms (when `getBranches` finishes). The submodule check alone adds **1.6 seconds** to startup.

### Auto-refresh fires immediately after initial load (+2770ms)

`sendInitialData()` calls `refreshVSCodeSourceControl()` which triggers `repo.status()` on the VS Code git API. This causes a git state change that the file watcher picks up after its 1s debounce, triggering `triggerAutoRefresh()` — a full re-fetch of all data. Since the initial load is still running when the watcher fires, it queues as pending, then executes immediately after the initial load completes.

Timeline:
```
00.683 — sendInitialData START (calls refreshVSCodeSourceControl)
02.739 — File watcher fires → triggerAutoRefresh queued as pending
03.554 — Initial load completes → pending auto-refresh starts immediately
03.555 — Full second data fetch begins (another 2770ms)
```

This auto-refresh exists to sync VS Code's Source Control panel after extension-initiated git operations. It is NOT needed on initial load because no git operations have been performed yet.

### Redundant frontend RPC calls

`App.tsx` useEffect at mount sends:
- `rpcClient.getSettings()` — already sent synchronously in `sendInitialData`
- `rpcClient.getSubmodules()` — already included in `initialData` payload; spawns 2 extra git processes

## Fix 1: Decouple submodules from initial data fetch

### Current behavior

`getSubmodules()` is one of 9 operations in `sendInitialData()`'s `Promise.allSettled` block (`src/WebviewProvider.ts`). It blocks the entire initial data payload from being sent until it completes. The submodule result is bundled into the `InitialDataPayload` and processed atomically by `setInitialData()` in the frontend store.

### Target behavior

1. **Fast-path in `GitSubmoduleService.getSubmodules()`**: Check `.gitmodules` existence with `fs.existsSync` before spawning any git processes. No file = return `ok([])` immediately.
2. **Remove from `Promise.allSettled`**: `getSubmodules` no longer participates in the main data fetch. The graph renders without waiting for submodule data.
3. **Fire separately**: After `sendInitialData()` posts the `initialData` message, fire submodule fetching as a separate non-blocking async call (like avatars). Send the result via the existing `submodulesData` response message type.
4. **Update `InitialDataPayload`**: Remove `submodules` and `submoduleStack` fields. Frontend `setInitialData()` no longer touches submodule state — it arrives separately via `setSubmodules()`.

### Implementation details

#### `src/services/GitSubmoduleService.ts`

Add `import * as path from 'path';` and `import { existsSync } from 'fs';` at the top.

In `getSubmodules()`, add a fast-path check before the `Promise.all`:

```ts
async getSubmodules(): Promise<Result<Submodule[]>> {
  // Fast-path: skip all git commands if no .gitmodules file exists
  const gitmodulesPath = path.join(this.workspacePath, '.gitmodules');
  if (!existsSync(gitmodulesPath)) {
    return ok([]);
  }

  const [statusResult, pathMap, urlMap] = await Promise.all([
    // ... existing code unchanged
  ]);
  // ... rest unchanged
}
```

#### `shared/messages.ts`

Remove `submodules` and `submoduleStack` from `InitialDataPayload`:

```ts

  totalLoadedWithoutFilter: number;
  hasMore: boolean;

  remotes: RemoteInfo[];
  authors: Author[];
  worktrees: WorktreeInfo[];
```

Also remove `Submodule` and `SubmoduleNavEntry` from the import if they are no longer used in this file (check — they are still used by `submodulesData` in `ResponseMessage`, so they stay in the import).

#### `src/WebviewProvider.ts` — `sendInitialData()`

**Remove from `Promise.allSettled`**: Remove the `submodulesSettled` destructured variable and the `traced('getSubmodules', this.gitSubmoduleService.getSubmodules())` line from the `Promise.allSettled` array. The array goes from 9 entries to 8.

**Remove the "Extract submodules" block** (~lines 665-674 in current code):

**Remove from payload construction** (~lines 742-743):
```ts

submodules,

```

**Fire submodules separately** — add after the `initialData` message post and after `loading: false`, similar to how avatars are handled:

```ts

if (isInitialLoad) {
  this.postMessage({ type: 'loading', payload: { loading: false } });
  this.initialLoadSent = true;
}

// Fetch submodules in background (non-blocking) — decoupled from initial data
// to avoid blocking graph render with slow `git submodule status`

// Fetch GitHub avatars in background (non-blocking), skip if avatars disabled

  void this.fetchAndSendGitHubAvatars(fetchedCommits);
}
```

**Add the `fetchAndSendSubmodules` private method** (near `sendSubmodulesData`):

```ts
/** Fetch submodules and send to webview as a non-blocking background operation. */
private async fetchAndSendSubmodules() {
  const result = await this.gitSubmoduleService.getSubmodules();
  if (result.success) {
    this.postMessage({
      type: 'submodulesData',
      payload: {
        submodules: result.value,
        stack: this.submoduleHandlers?.getStack() ?? [],
      },
    });
  }
  // Silently ignore errors — submodule data is non-critical for graph display
}
```

Note: The existing `sendSubmodulesData()` method can remain as-is since it's called from `handleMessage` for the `getSubmodules` RPC request. Or, it can be refactored to call `fetchAndSendSubmodules()` internally to avoid duplication:

```ts
private async sendSubmodulesData() {
  await this.fetchAndSendSubmodules();
}
```

But `sendSubmodulesData` also posts an error on failure. Choose whichever approach the implementer prefers — the key point is that both the initial-load path and the RPC-request path use the same `submodulesData` message type.

No replacement needed — the existing `setSubmodules()` action handles submodule state when the `submodulesData` message arrives. The store initializes with `submodules: []` and `submoduleStack: []` by default (already set in the store's initial state).

## Fix 2: Skip `refreshVSCodeSourceControl()` on initial load

### Current behavior

`sendInitialData()` always calls `refreshVSCodeSourceControl()` unless `isAutoRefresh=true`. On initial load, this triggers a cascading file watcher event that causes a full redundant re-fetch.

### Target behavior

Skip `refreshVSCodeSourceControl()` when `isInitialLoad` is true. The purpose of this call is to sync VS Code's SCM panel after extension-initiated git operations — on initial load, no git operations have been performed, so there's nothing to sync.

### Implementation details

#### `src/WebviewProvider.ts` — `sendInitialData()`

Change the guard condition. The `isInitialLoad` variable is computed later in the method, so move the check after that computation. Alternatively, use the existing `this.initialLoadSent` flag directly:

**Current code** (near the top of `sendInitialData`):
```ts
if (!isAutoRefresh) {
  this.refreshVSCodeSourceControl();
}
```

**Change to**:
```ts
if (!isAutoRefresh && this.initialLoadSent) {
  this.refreshVSCodeSourceControl();
}
```

`this.initialLoadSent` is `false` on the very first load and set to `true` after the initial `initialData` message is posted. This means:
- First load: skips refresh (no git ops have happened)
- Subsequent `sendInitialData` calls triggered by user actions (checkout, merge, etc.): `initialLoadSent` is `true`, refresh runs normally
- Auto-refreshes: already skipped by the `!isAutoRefresh` guard

## Fix 3: Remove redundant frontend RPC calls

### Current behavior

`App.tsx` useEffect at mount sends `rpcClient.getSettings()` and `rpcClient.getSubmodules()`. Both trigger backend `handleMessage` processing and git process spawning, duplicating work already done by `sendInitialData()`.

### Target behavior

Remove both calls.

### Implementation details

#### `webview-ui/src/App.tsx`

**Current code**:
```ts
useEffect(() => {
  rpcClient.initialize();
  rpcClient.getSettings();
  rpcClient.getSubmodules();
  rootRef.current?.focus();
}, []);
```

**Change to**:
```ts
useEffect(() => {
  rpcClient.initialize();
  rootRef.current?.focus();
}, []);
```

Settings are already sent synchronously by `sendInitialData()` via `this.sendSettingsData(settings)` before the parallel fetch begins. Submodules will now arrive via the decoupled `fetchAndSendSubmodules()` background call (Fix 1).
